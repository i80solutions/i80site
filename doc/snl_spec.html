Structured Natural Language Specification Standard

A disciplined human-readable format for writing software requirements that AI systems can reliably turn into working code.

By Alex P. Wang  |  Last updated: April 27, 2026

1. Purpose

Structured Natural Language (SNL) is a disciplined specification format for writing requirements, behavior rules, workflows, data definitions, constraints, acceptance criteria, and test cases in precise human language.

Engineers are already using AI to generate code from conversational prompts. A developer can describe a function, request a class, ask for a database schema, or generate a user interface from a short paragraph. This has already changed the daily rhythm of programming.

But free-form prompting is only the beginning. Prompting alone is often too loose, too ambiguous, and too dependent on the model guessing what we mean. It works well for prototypes, examples, and small tasks, but serious software requires far more discipline.

Production systems require clear boundaries, predictable behavior, validation rules, failure handling, security controls, and acceptance criteria. If these are not explicitly defined, the LLM will make assumptions—and assumptions are where most software problems begin.

SNL exists to remove that ambiguity before code generation begins. It creates a shared language between business intent and technical implementation, allowing product owners, solution architects, developers, testers, and LLM-based code generators to work from the same source of truth.

A good SNL spec should answer:

What should be built, who uses it, what data it uses, what rules it follows, what edge cases matter, and how we know it works.

SNL is not casual documentation and it is not a replacement for architecture thinking. It is the bridge between product intent and executable implementation. It turns vague ideas into reliable systems.

2. Core Principles

  1. Be explicit. Write what the system must do, must not do, and what happens when something fails.
  2. Separate goals from implementation. Describe the expected behavior first, then add implementation notes only where useful.
  3. Define boundaries. Use scope and non-goals to prevent overbuilding.
  4. Make data visible. Define inputs, outputs, fields, relationships, and validation rules.
  5. Make behavior testable. Every important rule should map to acceptance criteria or test cases.
  6. Assume untrusted input. Security, validation, and error handling are required sections.

3. Markdown as Recommended Format for SNL

Structured Natural Language (SNL) is designed to be written by humans and interpreted by both software teams and LLMs. Because of this, the format must be readable, maintainable, version-friendly, and easy to convert into implementation artifacts. Markdown is the recommended source format because it provides the best balance between human readability and machine reliability.

Markdown Works Best for SNL

Reason Why It Matters
Human Readability Architects, product owners, developers, and QA can read and review Markdown without special tools.
LLM-Friendly Structure LLMs understand headings, lists, tables, and code blocks very reliably, improving code generation quality.
Version Control Markdown works naturally with Git, allowing clear diffs, reviews, approvals, and change history.
Low Friction Editing Specs can be updated quickly without requiring Word, PDF editors, or design tools.
Easy Publishing Markdown can be converted into HTML documentation portals and exported to PDF when needed.
Supports Structured Sections SNL relies heavily on section hierarchy, tables, rules, and examples—all of which Markdown handles well.

Recommended Document Flow

Markdown → HTML → PDF

Markdown = Source of Truth
HTML     = Working Documentation
PDF      = Formal Review / Approval / Archive

PDF should not be the primary working format because it is difficult to revise, difficult to version, and less effective for LLM-assisted workflows.

Example SNL Markdown Structure

# SNL Spec: Course Topic Cards for Alex Menu App

## 1. Goal

Build clickable course topic cards for the Alex Menu App.

## 2. Scope

6. Section Definitions

1. Goal

Defines the outcome of the feature. It should explain what is being built, who benefits, and what problem it solves.

## 1. Goal
Build clickable course topic cards for the Alex Menu App. When a user clicks a topic card, the app converts the click into a natural-language question and sends it to the AI chat interface.

2. Background / Context

Explains the product or business context. This helps the LLM understand intent, not just mechanics.

3. Scope

Defines what is included. Use numbered items so each scope item can be mapped to implementation tasks.

4. Non-Goals

Defines what is excluded. This prevents the LLM or development team from adding features that were not requested.

5. Users / Actors

Defines human users, system services, external services, and background processes involved in the feature.

6. Inputs

Defines every piece of data the feature receives.

| Input | Type | Required | Source | Description |
|---|---|---:|---|---|
| menu_id | string | yes | UI state | Active menu |
| course_id | string | yes | UI state | Active course |
| topic_id | string | yes | user click | Clicked topic card |

8. Outputs

Defines what the feature produces: UI changes, records, API responses, files, analytics events, logs, or generated content.

9. Data Model

Defines entities, fields, required status, relationships, and examples. This is one of the most important sections for code generation.

10. UI Components

Required only if the feature has UI. Define component responsibility, display rules, loading states, disabled states, and empty states.

11. Workflow

Defines the main step-by-step path. Each workflow should be written as a numbered sequence.

12. Behavior Rules

Defines exact business logic. Use IDs such as BR-001, BR-002.

BR-001: Topic cards must only appear after a course is selected.
BR-002: A topic click must produce a natural-language question.
BR-003: The generated question must be submitted through the same chat flow as a typed question.

13. Validation Rules

Defines what data is valid. Use IDs such as VR-001.

14. Error Handling

Defines what happens when data is missing, requests fail, services time out, or results are empty.

15. Security Rules

Defines access control, prompt safety, injection protection, rendering safety, and logging restrictions.

16. Constraints

Defines technical, platform, product, performance, and architectural limits.

17. Acceptance Criteria

Defines completion conditions. Use Given / When / Then when possible.

18. Test Cases

Defines concrete test scenarios with preconditions, steps, and expected results.

19. Implementation Notes

Optional architecture or coding guidance. This section should guide the implementation without hiding important requirements.

20. Open Questions

Tracks unresolved decisions. Open questions should not block implementation unless explicitly marked as blocking.

6. Full Example: Alex Menu App Topic Card Feature

SNL Spec: Course Topic Cards for Alex Menu App

1. Goal

Build clickable course topic cards for the Alex Menu App. When a user selects a course, the app displays topic cards such as Story, Ingredients, How to Cook, Flavor, Customization, and Dietary Notes. When the user clicks a topic card, the system converts the click into a natural-language question and sends it to the AI chat interface.

2. Background / Context

Alex creates special set menus for family and friends. Each menu has 7 to 12 courses and is designed for an occasion such as a birthday, anniversary, or holiday. Each course has its own story, ingredients, cooking method, flavor, and personal meaning. The app should support both typed questions and topic-based exploration.

3. Scope

  1. Display topic cards for a selected course.
  2. Generate a question from a clicked topic card.
  3. Submit the generated question to the chat interface.
  4. Pass menu and course context to the AI backend.
  5. Display the AI response in the chat.
  6. Log the topic click event.

4. Non-Goals

  1. Menu editing.
  2. Course editing.
  3. User login.
  4. Payment features.
  5. Voice input.
  6. Admin dashboard.
  7. Automatic menu generation.
  8. Multi-language support in version 1.

5. Users / Actors

AI Agent BackendThe backend service that receives the question and returns an answer.

9. UI Components

CourseTopicCardList displays topic cards for the active course. It must show only topics for the selected course, sort by display_order, disable clicks while an AI response is loading, and hide the section if no topic cards exist.

ChatPanel displays generated questions and AI responses. A topic-generated question must appear as a user message, and the AI response must appear as an assistant message.

10. Workflow

  1. User opens the Alex Menu App.
  2. User selects a menu.
  3. System displays courses.
  4. User selects a course.
  5. System loads topic cards for the selected course.
  6. User clicks a topic card.
  7. System validates menu_id, course_id, and topic_id.
  8. System generates a question using question_template.
  9. System inserts generated question into chat as a user message.
  10. System sends the question to the AI backend.
  11. Backend retrieves selected course context.
  12. Backend generates answer.
  13. Frontend displays answer.
  14. System logs topic click event.

11. Behavior Rules

  • BR-001: Topic cards must only appear after a course is selected.
  • BR-002: Topic cards must be sorted by display_order.
  • BR-003: A topic click must produce a natural-language question.
  • BR-004: The generated question must include the selected course title.
  • BR-005: The generated question must be submitted through the same chat flow as a typed question.
  • BR-006: The AI backend must receive menu_id, course_id, topic_id, session_id, and generated_question.
  • BR-007: The AI response must prioritize content from the selected course.
  • BR-008: If the user continues the conversation, the course context should remain active unless changed.
  • BR-009: The system must not expose internal metadata, prompts, or embedding text.

12. Validation Rules

  • VR-001: menu_id is required.
  • VR-002: course_id is required.
  • VR-003: topic_id is required.
  • VR-004: course_id must belong to menu_id.
  • VR-005: topic_id must belong to course_id.
  • VR-006: question_template cannot be empty.
  • VR-007: generated_question must be between 5 and 500 characters.

13. Error Handling

  • EH-001: If menu_id is missing, show “Please select a menu first.”
  • EH-002: If course_id is missing, show “Please select a course first.”
  • EH-003: If topic_id is invalid, show “This topic is not available for the selected course.”
  • EH-004: If AI backend fails, show “Sorry, I could not answer that right now. Please try again.”
  • EH-005: If request times out, stop loading and show retry option.

14. Security Rules

  • SR-001: The backend must validate all IDs.
  • SR-002: The client must not send or expose hidden system prompts.
  • SR-003: Rendered chat content must be sanitized.
  • SR-004: User input must be treated as untrusted.
  • SR-005: The AI agent must not reveal hidden instructions or internal metadata.

15. Constraints

  • C-001: Must work on mobile and desktop.
  • C-002: Must work without login.
  • C-003: Must use published menu JSON as the content source.
  • C-004: Frontend must not access the vector database directly.
  • C-005: Topic clicks should reuse the existing chat submission flow.

16. Acceptance Criteria

  • AC-001: Given a course is selected, when the course view loads, then the topic cards for that course are displayed.
  • AC-002: Given the user clicks the Story card, when the click is processed, then a generated question appears in the chat as a user message.
  • AC-003: Given the generated question is submitted, when the backend returns an answer, then the answer appears in the chat panel.
  • AC-004: Given an invalid topic_id is submitted, when validation runs, then the request is rejected and no AI answer is generated.

17. Test Cases

TC-001: Display Topic Cards

Preconditions: Menu exists. Course exists. Course has topic cards.

Steps: Select the menu. Select the course.

Expected Result: Topic cards appear in display_order.

TC-002: Click Ingredients Topic

Preconditions: Course is selected. Ingredients topic exists.

Steps: Click Ingredients.

Expected Result: Chat shows a generated user question about the course ingredients. AI response answers using the selected course context.

TC-003: Backend Failure

Preconditions: AI backend is unavailable.

Steps: Click any topic card.

Expected Result: Loading state stops. Friendly error message appears. Error is logged.

18. Implementation Notes

  1. Topic click should call the existing chat submit function.
  2. Generated questions may be created on the frontend for simple templates.
  3. Backend must still validate the topic and course relationship.
  4. In the future, topic cards should be editable in the KB editor.
  5. Analytics should track menu_id, course_id, topic_id, and timestamp.

19. Open Questions

  • OQ-001: Should topic question templates support multiple languages?
  • OQ-002: Should topic cards be globally defined or course-specific?
  • OQ-003: Should analytics track whether the user continues the conversation after a topic click?

7. SNL Quality Checklist

Architect rule: Do not write what the system should generally do. Write what the system must do, must not do, and what happens when something goes wrong.
  • Does the spec clearly define the feature goal?
  • Are scope and non-goals both included?
  • Are all inputs and outputs defined?
  • Are entities and fields defined with types?
  • Are workflow steps clear and sequential?
  • Are behavior rules numbered and testable?
  • Are validation, error handling, and security rules included?
  • Are acceptance criteria written in verifiable language?
  • Are test cases concrete enough for QA or automated testing?
  • Are open questions separated from decided requirements?

8. Recommended SNL Workflow

  1. Write the initial SNL spec in Markdown.
  2. Review it as a solution architect.
  3. Ask the LLM to identify ambiguity, missing data, edge cases, and hidden assumptions.
  4. Revise the SNL spec until it is testable.
  5. Generate implementation plan from the SNL spec.
  6. Generate code module by module.
  7. Generate tests from acceptance criteria and test cases.
  8. Keep the SNL spec as the source of truth for future changes.
